# Lecture 3 — The Rest: Vulnerable Components, Logging, SSRF, Integrity

> *Four categories. The first is the one your build pipeline can prevent if you let it. The second is the one your incident responder will curse you for skipping. The third is the one that turns "the server fetches a URL" into "the server reads cloud credentials." The fourth is the one that makes pickle the most-feared standard-library import in Python.*

```
┌─────────────────────────────────────────────────────────────────────┐
│  AUTHORIZED USE ONLY                                                │
│                                                                     │
│  The vulnerable code in this lecture is for the lab app and your    │
│  own machine only. Do not deploy any of these snippets to a public  │
│  service. Do not run any of the attack payloads against a service   │
│  you do not operate.                                                │
└─────────────────────────────────────────────────────────────────────┘
```

This lecture closes the OWASP Top 10 2021 walk:

- **`A06:2021 Vulnerable and Outdated Components`** (was `A09:2017`, kept because the bug ships with `pip install` and your CI pipeline is the only place it can be stopped cheaply).
- **`A09:2021 Security Logging and Monitoring Failures`** (was `A10:2017`, kept because *not knowing you were attacked* turns a containable incident into a regulatory breach).
- **`A10:2021 Server-Side Request Forgery (SSRF)`** (new top-10 entry in 2021, added by community survey because cloud metadata and internal-network access made SSRF a top-tier vulnerability).
- **`A08:2021 Software and Data Integrity Failures`** (new in 2021, naming the supply-chain class that the 2017 list under-served, including `pickle` and JWT integrity).

The order in the lecture is the order in which you encounter them in a code review: components in the lockfile, logging at the call sites, SSRF at the fetch sites, integrity at the deserialise sites.

## 1. `A06:2021 Vulnerable and Outdated Components`

### 1.1 What OWASP says

*"You are likely vulnerable: If you do not know the versions of all components you use (both client-side and server-side). This includes components you directly use as well as nested dependencies. If software is vulnerable, unsupported, or out of date. This includes the OS, web/application server, database management system, applications, APIs and all components, runtime environments, and libraries. If you do not scan for vulnerabilities regularly and subscribe to security bulletins related to the components you use. If you do not fix or upgrade the underlying platform, frameworks, and dependencies in a risk-based, timely fashion."* (Top 10 2021, `A06:2021`, condensed.)

CWE families:

- **CWE-1104** Use of Unmaintained Third Party Components
- **CWE-1395** Dependency on Vulnerable Third-Party Component (added 2024)
- **CWE-937** OWASP Top 10 2013: Using Components with Known Vulnerabilities (parent)

### 1.2 The Python supply chain

A Python application has three layers of dependency:

1. **Your direct dependencies.** What is in `requirements.txt` or `pyproject.toml`.
2. **Transitive dependencies.** What your direct dependencies depend on. The pin-fan-out — a top-level Flask app may pull in 40+ packages.
3. **System dependencies.** What `pip install` calls out to via C extensions — `libssl`, `libxml2`, `zlib`. These are the *operating system's* supply chain and surface as Docker base-image vulnerabilities.

For each, the question is: do you know what version is installed, and do you know whether that version is vulnerable?

### 1.3 Lockfiles and `pip-tools`

**Anti-pattern (`requirements_bad.txt`):**

```
Flask
SQLAlchemy
requests
```

Unpinned. Every `pip install` produces a different dependency tree. CI passes today; CI fails tomorrow because a transitive dep released a breaking change overnight.

**Fixed (`requirements.in` + `requirements.txt`):**

```
# requirements.in — what you actually want
Flask
SQLAlchemy
requests
```

```
# requirements.txt — generated by `pip-compile requirements.in`
# pinned to specific versions with hashes
flask==3.0.3 \
    --hash=sha256:... \
    --hash=sha256:...
...
```

`pip-tools` (`pip install pip-tools`) gives you `pip-compile` (to generate the lockfile) and `pip-sync` (to install exactly what is in the lockfile, removing anything else). The lockfile is committed; CI and production install from it; supply-chain *drift* becomes impossible.

The same goal in `uv` (the fast Rust-based pip alternative; widely adopted in 2024-2025): `uv pip compile requirements.in -o requirements.txt`. The interface is similar.

### 1.4 `pip-audit`

`pip-audit` (PyPA, `pip install pip-audit`) is the official supply-chain scanner.

```bash
# Scan the current environment
pip-audit

# Scan a requirements file
pip-audit -r requirements.txt

# Fail CI on findings
pip-audit -r requirements.txt --strict
```

It queries the **OSV** database (<https://osv.dev/>) for known vulnerabilities matching each installed version and prints CVEs and fix versions. Run it on every PR.

Alternative: **`safety`** (PyUp; `pip install safety`). Similar interface, different vulnerability source. Either is acceptable; running both is fine; running neither is `A06`.

### 1.5 `dependabot`, `renovate`, automated PRs

The automated half of the story: a bot that opens PRs to bump versions when an upstream release is available, and tests pass.

- **GitHub Dependabot** is on by default for public repos and free for private. Enable it via `.github/dependabot.yml`.
- **Renovate** (now part of Mend) is more configurable, supports more ecosystems, runs as a GitHub App.
- **`pyup`, `safety-ci`** are commercial alternatives with similar shapes.

The choice matters less than *having one*. A repo that lets dependencies drift for 18 months is one CVE away from a P1.

### 1.6 Base image vulnerabilities

If you ship via Docker, your base image is part of your supply chain. A `python:3.11-bullseye` image inherits Debian Bullseye's package set; a Bullseye CVE is your CVE.

The mitigation hierarchy, easiest first:

1. **Pin the base by digest, not tag.** `python:3.11-slim@sha256:...` survives the upstream re-tagging the floating tag for the next point release.
2. **Rebuild on a schedule** (weekly is normal; daily for high-security services) so that base-image CVE patches reach your image.
3. **Scan with `trivy`** or **`grype`** in CI: `trivy image myorg/myapp:latest`. Either tool will list CVEs by severity, with patched-in versions.
4. **Choose a minimal base.** `python:3.11-slim`, `cgr.dev/chainguard/python`, or a distroless image cuts the attack surface — fewer packages, fewer CVEs.

### 1.7 The post-XZ-Utils world

The **XZ Utils backdoor (CVE-2024-3094)**, disclosed March 2024 by Andres Freund, shifted the supply-chain conversation. A maintainer who had earned commit access over two years committed a backdoor into a library used by `sshd` indirectly via `liblzma`. Detection was accidental — Freund noticed `sshd` was unusually slow.

Lessons baked into the 2025 release candidate:

- **Maintainer identity is now in scope.** A package "by `JiaT75`" with one commit is a different supply-chain risk than a package by 50 humans across 10 years.
- **Build provenance matters.** Sigstore (`cosign`, `sigstore-python`) and PEP 740 (when finalised) move toward signed artifacts whose build provenance is verifiable.
- **The "test fixture" pattern** — a compressed test fixture that contains a backdoor extracted at build time — is a known pattern now. Scrutinise binary blobs in source trees.

### 1.8 Detection

- **`pip-audit -r requirements.txt --strict`** in CI.
- **`pip list --outdated`** for the easy "what's old" check.
- **GitHub Advisory Database** subscription for your dependency set.
- **`trivy fs .`** for repos that include lockfiles in multiple ecosystems.

## 2. `A09:2021 Security Logging and Monitoring Failures`

### 2.1 What OWASP says

*"Insufficient logging, detection, monitoring, and active response occurs any time: Auditable events, such as logins, failed logins, and high-value transactions, are not logged. Warnings and errors generate no, inadequate, or unclear log messages. Logs of applications and APIs are not monitored for suspicious activity. Logs are only stored locally. Appropriate alerting thresholds and response escalation processes are not in place or effective. Penetration testing and scans by dynamic application security testing (DAST) tools do not trigger alerts. The application cannot detect, escalate, or alert for active attacks in real time or near real time."* (Top 10 2021, `A09:2021`, condensed.)

CWE families:

- **CWE-778** Insufficient Logging
- **CWE-117** Improper Output Neutralisation for Logs (log injection)
- **CWE-223** Omission of Security-relevant Information
- **CWE-532** Insertion of Sensitive Information into Log File

### 2.2 What to log

The **OWASP Logging Cheat Sheet** lists the minimum:

- **Authentication events** — successes and failures, with the user ID, source IP, user agent, result, reason. Never the password.
- **Authorisation failures** — user, resource, action attempted, result.
- **Session-management events** — session created, session destroyed, session refresh, session revoked.
- **Administrative actions** — role changes, account creation, password resets initiated, audit-log access.
- **High-value transactions** — payments, transfers, account-deletion, data export, MFA changes.
- **Input-validation rejections** at the application boundary — they reveal probing.
- **Application errors** — at a granularity that does not flood the log but does not hide the unusual.

### 2.3 What never to log

- **Passwords, password reset tokens, full session cookies.**
- **Full payment-card numbers (PANs).** Truncate to the last four; if you need more, see PCI DSS §3.4. *And* tokenise at the boundary so the full PAN is never in your application logs in the first place.
- **Full social-security numbers, government IDs.** Truncate.
- **API keys, OAuth tokens, JWT tokens** in full. Log the JWT's `jti` if you need a correlator.
- **PII** beyond what regulatory framing allows. GDPR, CCPA, and HIPAA each have specific rules; comply with the strictest applicable.

### 2.4 Structured logs, not strings

**Vulnerable (`log_bad.py`):**

```python
import logging

log = logging.getLogger(__name__)

def login(username, ip):
    log.info(f"login attempt by {username} from {ip}")
```

Two failures: the log is a string that is hard to query (grep is not analysis); and a username of `bob\n[CRITICAL] system compromised` injects a fake log line (CWE-117, log injection).

**Fixed (`log_good.py`):**

```python
import logging
import structlog

structlog.configure(
    processors=[
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso", utc=True),
        structlog.processors.JSONRenderer(),
    ],
)
log = structlog.get_logger()

def login(username, ip, result, reason=None):
    log.info(
        "auth.login",
        username=username,
        source_ip=ip,
        result=result,           # "success" | "fail"
        reason=reason,           # "bad_password" | "account_locked" | None
    )
```

The output is one JSON object per line:

```json
{"timestamp": "2025-05-13T12:34:56Z", "level": "info", "event": "auth.login",
 "username": "alice", "source_ip": "203.0.113.7", "result": "fail", "reason": "bad_password"}
```

Queryable from any log aggregator. Log injection closed because the username is a *field*, not a string concatenation. The `event` name is the *stable identifier* for the log line; downstream systems alert on `event=auth.login AND result=fail`.

Follow **Elastic Common Schema (ECS)** field names where possible — `source.ip`, `user.name`, `event.action`, `event.outcome` — so your logs interoperate with any ECS-aware aggregator (Elastic, OpenSearch, Splunk with the ECS app).

### 2.5 Monitoring and alerting

Logs that nobody reads are not monitoring. The alerting layer is what turns logs into detection:

- **Failed-login rate** above N per minute across the system → page on call.
- **Authorisation failures from a single account or IP** above threshold → security alert.
- **5xx rate** above baseline → engineering alert (also covers many DoS shapes).
- **Admin actions outside business hours** → security audit signal.
- **First-time-from-this-IP for a privileged user** → soft alert (often a VPN change; sometimes account takeover).

For a small team, the minimum is: structured JSON logs → a free-tier aggregator (Grafana Loki, Better Stack free, self-hosted Vector + Loki) → three to five alerts in the aggregator's rule engine. The work is *picking the alerts*, not the infrastructure.

### 2.6 Retention and forensics

For incident response, you need logs going back at least as far as the typical *dwell time* of an undetected intruder. Industry averages (per Mandiant M-Trends, IBM Cost of a Data Breach) hover at **dozens to hundreds of days** for unauthenticated network attackers, longer for insiders.

A reasonable default:

- **30 days hot** — query latency under a second, full retention.
- **365 days cold** — S3 / object-store archive, queryable on demand, slow.
- **Indefinitely** for audit-relevant events (admin actions, payment events, account deletions) per regulatory framing.

### 2.7 Detection

- **`semgrep`** rules in `p/owasp-top-ten` flag `log(... + user_input + ...)` patterns.
- **`grep`** your codebase for `logging.info(f"...")` — most of these are A09 latent (unstructured, often un-redacted).
- **Audit your own logs** with a fake credential-stuffing run — does anything alert?

## 3. `A10:2021 Server-Side Request Forgery (SSRF)`

### 3.1 What OWASP says

*"SSRF flaws occur whenever a web application is fetching a remote resource without validating the user-supplied URL. It allows an attacker to coerce the application to send a crafted request to an unexpected destination, even when protected by a firewall, VPN, or another type of network access control list (ACL). As modern web applications provide end-users with convenient features, fetching a URL becomes a common scenario. As a result, the incidence of SSRF is increasing. Also, the severity of SSRF is becoming higher due to cloud services and the complexity of architectures."* (Top 10 2021, `A10:2021`.)

CWE: **CWE-918** Server-Side Request Forgery.

SSRF was added to the Top 10 in 2021 *by community survey*, not by data alone — the community recognised that the cloud-metadata-exfiltration class had become severe enough to warrant a dedicated category.

### 3.2 The textbook SSRF

A "URL preview" feature — paste a URL, the server fetches it and renders the title.

**Vulnerable (`ssrf_bad.py`):**

```python
import requests
from flask import Flask, request

app = Flask(__name__)

@app.route("/preview")
def preview():
    url = request.args.get("url", "")
    # No validation. The server fetches whatever the user asks.
    r = requests.get(url, timeout=5)
    return r.text[:1024]
```

Attacker payloads:

- `?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/` — AWS instance metadata; on IMDSv1 instances this returns IAM credentials. Game over on AWS.
- `?url=http://localhost:6379/` — Redis on the same host, often unauthenticated.
- `?url=http://10.0.0.0:8080/admin/` — internal admin panel.
- `?url=file:///etc/passwd` — `requests` will not follow `file://` by default in current versions, but older versions and other clients may.

### 3.3 The fix — allow-list, never deny-list

**Fixed (`ssrf_good.py`):**

```python
import ipaddress
import socket
from urllib.parse import urlparse
import requests
from flask import Flask, request

app = Flask(__name__)

ALLOWED_SCHEMES = {"http", "https"}
ALLOWED_PORTS = {80, 443}

def is_safe_url(url: str) -> bool:
    parsed = urlparse(url)
    if parsed.scheme not in ALLOWED_SCHEMES:
        return False
    if parsed.port is not None and parsed.port not in ALLOWED_PORTS:
        return False
    if not parsed.hostname:
        return False
    # Resolve to *all* addresses; reject if any is private, loopback, link-local, or reserved.
    try:
        infos = socket.getaddrinfo(parsed.hostname, parsed.port or 443, proto=socket.IPPROTO_TCP)
    except socket.gaierror:
        return False
    for family, _, _, _, sockaddr in infos:
        ip = ipaddress.ip_address(sockaddr[0])
        if ip.is_private or ip.is_loopback or ip.is_link_local or ip.is_reserved or ip.is_multicast:
            return False
        # AWS metadata explicitly:
        if str(ip) == "169.254.169.254":
            return False
    return True

@app.route("/preview")
def preview():
    url = request.args.get("url", "")
    if not is_safe_url(url):
        return ("bad url", 400)
    # Disable redirect following: a 302 to 169.254.169.254 bypasses the pre-check otherwise.
    r = requests.get(url, timeout=5, allow_redirects=False)
    return r.text[:1024]
```

The defences are stacked because SSRF has many bypasses:

- **Scheme allow-list.** `http`, `https`. Not `file`, not `gopher`, not `ftp`.
- **Port allow-list.** `80`, `443`. Not `22` (internal SSH banners), not `6379` (Redis), not arbitrary high ports.
- **DNS rebinding defence by re-resolving.** Resolve at validation time, then *pass the resolved IP, not the hostname*, to the HTTP client — otherwise an attacker-controlled DNS server can resolve to a public IP at validation and to `127.0.0.1` at fetch.
- **Block private IP ranges.** RFC 1918, loopback, link-local, multicast, reserved.
- **Explicitly block IMDS.** `169.254.169.254` and the GCP/Azure equivalents.
- **`allow_redirects=False`.** A 302 from a legitimate server to an attacker URL bypasses the pre-check.
- **Egress firewall at the network layer.** Belt-and-braces. The application's egress to anything other than the documented allow-list is denied at the network. SSRF that bypasses the application check still cannot reach metadata.

### 3.4 IMDSv2 — the AWS-side fix

On AWS, enable **IMDSv2** on every EC2 instance:

```bash
aws ec2 modify-instance-metadata-options \
    --instance-id i-0123456789abcdef0 \
    --http-tokens required \
    --http-put-response-hop-limit 1
```

IMDSv2 requires a session token obtained via a `PUT /latest/api/token` request with a `X-aws-ec2-metadata-token-ttl-seconds` header — a request shape that a generic `requests.get` will not produce, and a hop-limit of 1 prevents container egress from reaching the metadata service. Enabled by default on instances launched from Amazon Linux 2023; *opt-in* on older instances. Verify with `aws ec2 describe-instances` and the `HttpTokens` field.

Equivalents on GCP and Azure:

- **GCP:** require the `Metadata-Flavor: Google` header (long-standing); also restrict metadata access by instance metadata policy.
- **Azure:** require `Metadata: true` header; use Managed Identities with scoped permissions.

### 3.5 Library quirks worth knowing

- **`requests`** follows redirects by default. `allow_redirects=False` is required for SSRF defence.
- **`httpx`** does not follow redirects by default. Slightly safer default; still check at the URL parser.
- **`urllib3`** sits under both. The `connect()` is where DNS resolution happens; intercepting there is the right place for a low-level allow-list.
- **`aiohttp`** has its own redirect handling; verify both client and connector configuration.

### 3.6 Detection

- **Outbound-connection logs from the application server.** A spike in connections to `169.254.169.254` is unmistakable.
- **Cloud audit logs.** AWS CloudTrail logs IAM credential issuance to the metadata service; a sudden new IAM principal acting from a strange IP is a red flag.
- **Application logs.** Log every URL fetched by the URL-preview endpoint, with the requester. Alert on private-range IPs in the resolved set.

## 4. `A08:2021 Software and Data Integrity Failures`

### 4.1 What OWASP says

*"Software and data integrity failures relate to code and infrastructure that does not protect against integrity violations. An example of this is where an application relies upon plugins, libraries, or modules from untrusted sources, repositories, and content delivery networks (CDNs). An insecure CI/CD pipeline can introduce the potential for unauthorised access, malicious code, or system compromise. Lastly, many applications now include auto-update functionality, where updates are downloaded without sufficient integrity verification and applied to the previously trusted application. Attackers could potentially upload their own updates to be distributed and run on all installations. Another example is where objects or data are encoded or serialised into a structure that an attacker can see and modify, vulnerable to insecure deserialisation."* (Top 10 2021, `A08:2021`, condensed.)

CWE families:

- **CWE-502** Deserialisation of Untrusted Data
- **CWE-829** Inclusion of Functionality from Untrusted Control Sphere
- **CWE-494** Download of Code Without Integrity Check
- **CWE-345** Insufficient Verification of Data Authenticity

### 4.2 `pickle` — the Python-specific A08

`pickle` is a *code-execution format*. The pickle stream can construct any Python object, including by importing arbitrary modules and calling arbitrary callables. Loading a pickle from an untrusted source is equivalent to running `exec` on attacker input.

**Vulnerable (`pickle_bad.py`):**

```python
import pickle
from flask import Flask, request

app = Flask(__name__)

@app.route("/cart", methods=["POST"])
def restore_cart():
    blob = request.get_data()
    # User-supplied bytes deserialised with pickle. RCE on the next line.
    cart = pickle.loads(blob)
    return f"loaded {len(cart)} items"
```

The attacker's pickle:

```python
import os
import pickle

class Exploit:
    def __reduce__(self):
        return (os.system, ("id; cat /etc/passwd > /tmp/leak",))

payload = pickle.dumps(Exploit())
# POST payload to /cart → RCE.
```

**Fixed:**

```python
import json
from flask import Flask, request

app = Flask(__name__)

@app.route("/cart", methods=["POST"])
def restore_cart():
    try:
        cart = json.loads(request.get_data())
    except json.JSONDecodeError:
        return ("bad json", 400)
    # Validate the schema explicitly (pydantic, marshmallow, or hand-rolled).
    if not isinstance(cart, list):
        return ("bad schema", 400)
    for item in cart:
        if not isinstance(item, dict) or "sku" not in item or "qty" not in item:
            return ("bad schema", 400)
    return f"loaded {len(cart)} items"
```

The rule is simple: **never pickle.loads on data you did not pickle.dumps yourself, in the same process, in the same trust boundary.** For client-server, pickle is wrong; use JSON (or msgpack, or protobuf) and validate the schema.

### 4.3 `yaml.load` — the same shape, in YAML

```python
import yaml

# Vulnerable: yaml.load with the default Loader can construct arbitrary Python objects.
data = yaml.load(user_supplied_bytes)  # DON'T

# Fixed: safe_load constructs only basic types — dict, list, str, int, float, bool, None.
data = yaml.safe_load(user_supplied_bytes)
```

`yaml.load` has been marked as requiring an explicit `Loader=` argument since PyYAML 5.1 (2019), but the lesson is recurring — never use the default loader on untrusted YAML.

### 4.4 JWT integrity (re-visited)

Covered in Lecture 2 §1.7 under A02. The A08 framing is complementary: A02 covers the *crypto* failures (weak algorithm, predictable secret); A08 covers the *integrity* failure of *trusting an unverified signature* in the first place. The fixes overlap:

- Pin the algorithm on the verifier (`algorithms=["HS256"]`).
- Verify *every* required claim (`exp`, `iat`, `iss`, `aud`).
- Reject `alg=none` explicitly.

### 4.5 CI/CD integrity

A08 names the CI pipeline as part of the integrity surface. The supply-chain concerns:

- **Build secrets in CI logs.** A misconfigured `echo $TOKEN` ships the token to the world for any public-fork build. Use the CI's masked-secret feature, and audit logs for leaks.
- **`curl | sh` installers in CI.** A 5-second window for the attacker to swap the script. Pin script versions by hash; use the package manager.
- **Reusable workflows / third-party actions.** A GitHub Action by `unknown/random-action@main` runs in your repo's privilege context. Pin to a commit SHA, not a tag; review the action source.
- **Artifact signing.** Sign your releases. Sigstore + `cosign` is the modern keyless path; PEP 740 (when finalised) brings this to PyPI.

### 4.6 Auto-update integrity

If your application updates itself, the update channel is a privileged path. The minimum checks:

- **TLS to the update server**, with cert pinning if practical.
- **Cryptographic signature on the update** — a release-signing key whose public counterpart ships with the application. Update fails closed if the signature does not verify.
- **Version monotonicity** — refuse to "update" to an older version (a *rollback attack* re-introduces patched vulnerabilities).
- **Update channel integrity** — the same key signs the index of releases, not just the artifacts.

### 4.7 Detection

- **`bandit`** flags `pickle.loads`, `yaml.load`, `eval`, `exec` on common patterns.
- **`semgrep`** rules in `p/owasp-top-ten` catch the common deserialisation shapes.
- **CI log audit** — search for `password`, `secret`, `token` in recent build logs; rotate anything that lands.
- **`gh attestation verify`** for GitHub-attested release artifacts (Sigstore-backed).

## 5. Summary — the closing four

| Category | Vulnerable pattern | Fix |
|---|---|---|
| `A06 Vulnerable and Outdated Components` | Unpinned deps; no scanner; stale base image | `pip-tools`/`uv` lockfile; `pip-audit` in CI; Dependabot/Renovate; weekly base-image rebuild; `trivy` scan |
| `A09 Logging and Monitoring Failures` | f-string logs; missing auth-event logs; no retention; no alerting | `structlog` JSON; log all auth + authz events + admin actions; ECS field names; 30 days hot / 365 days cold; alert rules |
| `A10 Server-Side Request Forgery (SSRF)` | `requests.get(user_url)` with no validation | Scheme + port + IP allow-list; resolve and pass IP; `allow_redirects=False`; egress firewall; IMDSv2 |
| `A08 Software and Data Integrity Failures` | `pickle.loads`; `yaml.load`; unverified JWT; unsigned releases; unpinned CI actions | JSON + schema; `yaml.safe_load`; pin algorithm and claims; Sigstore; pin CI actions by SHA |

## 6. From here to the mini-project

You have now walked all ten OWASP Top 10 2021 categories with Python-specific examples and fixes. The exercises this week ask you to run a single category at a time on your own machine. The mini-project asks you to patch *all ten* in a deliberately vulnerable Flask app and ship the patches as a portfolio repo.

Two reading reminders before you start the exercises:

- **Always check the 2025 release candidate page** before you frame a finding in 2025+. The category numbers, names, and groupings change. Cite 2021 IDs *and* the corresponding 2025 ID where the 2025 RC has stabilised. The 2025 final is expected mid-2025; the page is the source of truth.
- **Cross-reference CWE and ATT&CK** when you write findings. A finding that cites `OWASP A03:2021 Injection / CWE-89 / MITRE T1190` is legible to security engineers, auditors, and incident responders alike. A finding that says "SQL injection" is legible only to people who already know.
